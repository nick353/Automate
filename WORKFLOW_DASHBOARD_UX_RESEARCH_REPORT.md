# Workflow Dashboard - UXリサーチレポート
## 世界展開レベルの改善提案

---

## Executive Summary

**現状評価（3行要約）**
Workflow Dashboardは、自然言語ブラウザ自動化という革新的なコンセプトと、ライブビュー機能という強力な差別化要素を持つプロダクトです。しかし、初見ユーザー体験の設計不足、多言語対応の欠如、アクセシビリティ基準への未対応により、世界展開には大きな改善が必要です。特に、サービス価値の即座の理解と、初回利用時の「Aha moment」までの導線が不明確です。

**最も重要な改善ポイント3つ**
1. **初見ユーザー体験の根本的再設計**：ランディングページの追加、明確な価値提案、オンボーディングフローの実装
2. **多言語対応とローカライゼーション**：i18nフレームワークの導入、主要市場（英語圏、中国、EU）への対応
3. **アクセシビリティとパフォーマンスの最適化**：WCAG 2.1 AA準拠、モバイルファーストデザイン、パフォーマンス最適化

## Phase 1: 初見ユーザー体験（First-Time User Experience）

### 1.1 トップページ評価

#### 現状の問題点

**問題1: ランディングページが存在しない**
- **影響度**: 高
- **現状**: `/login`ページが最初の画面で、サービス内容が全く説明されていない
- **問題**: 初見ユーザーは「このサービスは何か」「自分に価値があるか」を判断できない
- **改善案**: 
  - 専用ランディングページ（`/`）を作成
  - ヒーローセクションに「自然言語でブラウザ自動化」という価値提案を明確に表示
  - 30秒デモ動画またはインタラクティブデモを配置
  - 「今すぐ始める」CTAボタンを目立つ位置に配置（オレンジ/緑のアクセントカラー）
- **参考事例**: 
  - [Figma](https://www.figma.com/) - ヒーローセクションで価値提案を明確に
  - [Notion](https://www.notion.so/) - ビジュアルとコピーで即座に理解できる

**問題2: ログインページでのサービス説明不足**
- **影響度**: 高
- **現状**: `Login.jsx`には「Workflow Dashboard」という名前とロゴのみ
- **問題**: 新規ユーザーがサービス内容を理解できない
- **改善案**:
  - ログインページの左側（デスクトップ）または上部（モバイル）にサービス説明を追加
  - 「自然言語でブラウザ操作を自動化」「動画から自動生成」「リアルタイム監視」の3つの主要機能をアイコン付きで表示
  - 「サインアップ」ボタンをより目立つデザインに（現在は小さなリンク）
- **参考事例**: 
  - [Shopify](https://www.shopify.com/login) - ログインページに価値提案を表示
  - [Linear](https://linear.app/) - シンプルだが価値が伝わるデザイン

**問題3: サービス名が抽象的**
- **影響度**: 中
- **現状**: 「Workflow Dashboard」という名前だけでは機能が不明確
- **改善案**:
  - サブタイトルを追加：「AIが理解する、自然言語ブラウザ自動化プラットフォーム」
  - または、より具体的な名前を検討：「BrowserBot」「AutoFlow」「TaskAI」など
- **参考事例**: 
  - 「Zapier」- 名前だけでなく「Automate your work」というサブタイトルで明確化

### 1.2 サインアップ/オンボーディング体験

#### 現状の問題点

**問題1: オンボーディングフローが存在しない**
- **影響度**: 高
- **現状**: サインアップ後、即座にダッシュボードに遷移。何をすべきか不明
- **問題**: 「Aha moment」までの時間が長すぎる。ユーザーが迷子になる
- **改善案**:
  - **ステップ1**: ウェルカムメッセージ + サービス紹介（30秒）
  - **ステップ2**: 最初の認証情報追加ガイド（Anthropic APIキーの追加）
  - **ステップ3**: サンプルタスクの作成またはデモタスクの実行
  - **ステップ4**: ライブビューの体験（デモ実行をリアルタイムで確認）
- **実装難易度**: 中
- **優先度**: P1
- **参考事例**: 
  - [Notion](https://www.notion.so/) - インタラクティブなオンボーディング
  - [Figma](https://www.figma.com/) - 段階的な機能紹介

**問題2: サインアップフォームの障壁**
- **影響度**: 中
- **現状**: `Signup.jsx`ではメールとパスワードのみ（良い点）
- **問題**: パスワード強度の視覚的フィードバックがない
- **改善案**:
  - パスワード入力中に強度インジケーターを表示（弱/中/強）
  - パスワード要件を明確に表示：「6文字以上、英数字を含む」
  - ソーシャルログイン（Google、GitHub）を追加
- **実装難易度**: 低
- **優先度**: P2
- **参考事例**: 
  - [Vercel](https://vercel.com/signup) - パスワード強度インジケーター
  - [Canva](https://www.canva.com/) - ソーシャルログイン対応

**問題3: 初回ログイン後の空の状態**
- **影響度**: 高
- **現状**: ダッシュボードが空の状態で、何をすべきか不明
- **問題**: ユーザーが次のアクションを取れない
- **改善案**:
  - 空の状態に「最初のタスクを作成」という大きなCTAボタン
  - 「動画から自動生成」と「手動で作成」の2つの選択肢を提示
  - クイックスタートガイドをモーダルで表示
- **実装難易度**: 低
- **優先度**: P1
- **参考事例**: 
  - [Notion](https://www.notion.so/) - 空の状態でのガイダンス
  - [Airtable](https://airtable.com/) - インタラクティブなチュートリアル

### 1.3 主要機能の操作性

#### 現状の問題点

**問題1: タスク作成の複雑さ**
- **影響度**: 高
- **現状**: `TaskForm.jsx`で自然言語プロンプトを直接入力する必要がある
- **問題**: ユーザーが「何を書けばいいか」わからない
- **改善案**:
  - プロンプト入力欄にプレースホルダーを追加：「例：楽天RMSにログインして、今日の売上データを取得して、CSVファイルに保存して」
  - よく使われるタスクのテンプレートを提供
  - AIアシスタントボタン：「タスクの目的を教えてください」→ AIがプロンプトを生成
- **実装難易度**: 中
- **優先度**: P1
- **参考事例**: 
  - [ChatGPT](https://chat.openai.com/) - プロンプト例の提示
  - [Zapier](https://zapier.com/) - テンプレートの提供

**問題2: エラーメッセージが技術的**
- **影響度**: 中
- **現状**: `TaskWizard.jsx`で「Upload failed. Please try again.」など、原因が不明
- **問題**: ユーザーが何を修正すべきかわからない
- **改善案**:
  - エラーメッセージを具体的に：「動画ファイルのサイズが500MBを超えています。より小さなファイルをアップロードしてください。」
  - 解決策を提示：「ファイルを圧縮する」「別の形式で試す」など
  - エラーコードではなく、ユーザーフレンドリーなメッセージ
- **実装難易度**: 低
- **優先度**: P2
- **参考事例**: 
  - [Stripe](https://stripe.com/) - 明確なエラーメッセージと解決策
  - [GitHub](https://github.com/) - ユーザーフレンドリーなエラー表示

**問題3: ライブビューの発見性が低い**
- **影響度**: 中
- **現状**: 実行中のタスクをクリックしないとライブビューにアクセスできない
- **問題**: 主要機能なのに見つけにくい
- **改善案**:
  - ダッシュボードの「Running Now」カードから直接ライブビューにリンク
  - 実行中のタスクカードに「ライブビューを見る」ボタンを追加
  - 通知バナー：「タスクが実行中です。ライブビューで確認する」
- **実装難易度**: 低
- **優先度**: P2
- **参考事例**: 
  - [Vercel](https://vercel.com/) - デプロイ中のリアルタイム表示
  - [Netlify](https://www.netlify.com/) - ビルドプロセスの可視化

## Phase 2: ペルソナ予測＆ニーズ分析

### Primary Persona（メインターゲット）

**ペルソナ名**: 「自動化に興味があるが技術的知識が限定的なマーケター/営業担当」

- **職業/役割**: 
  - 中小企業のマーケティング担当者
  - 営業担当者
  - データアナリスト（非エンジニア）

- **解決したい課題（Jobs to be Done）**:
  - 「毎日のデータ収集作業を自動化したい」
  - 「複数のサイトから情報を取得してレポートを作成したい」
  - 「手動で繰り返している作業を減らしたい」

- **利用シーン**:
  - 業務時間中、デスクトップで使用
  - 朝のルーティン作業の自動化
  - 週次/月次レポートの自動生成

- **技術リテラシーレベル**: 中程度
  - Excel、Googleスプレッドシートは使える
  - プログラミングはできない
  - APIの概念は理解していない

- **意思決定における重視ポイント**:
  - **使いやすさ**: 直感的な操作
  - **コストパフォーマンス**: 月額$20-50程度
  - **信頼性**: エラーが少ない
  - **サポート**: 問題が起きたときに助けが得られる

### Secondary Persona（サブターゲット）

**ペルソナ名**: 「効率化を重視するフリーランス/起業家」

- **職業/役割**:
  - フリーランスのデザイナー/コンサルタント
  - スタートアップの創業者
  - 個人事業主

- **解決したい課題（Jobs to be Done）**:
  - 「クライアント向けの定期レポートを自動化したい」
  - 「複数のサービスを連携させたい」
  - 「時間を節約して本業に集中したい」

- **利用シーン**:
  - 自宅やカフェでの作業中
  - モバイルでも確認したい（実行はデスクトップ）
  - 夜間の自動実行

- **技術リテラシーレベル**: 高い
  - 基本的なプログラミング知識あり
  - APIの概念を理解している
  - 新しいツールに慣れるのが早い

- **意思決定における重視ポイント**:
  - **柔軟性**: カスタマイズ可能
  - **機能の豊富さ**: 様々な連携が可能
  - **価格**: 月額$50-100程度まで許容
  - **スピード**: 素早く設定できる

### 現状のペルソナ設計への指摘

**問題1: ターゲットが広すぎる**
- **現状**: マーケターから開発者まで幅広い層を想定している
- **問題**: メッセージがぼやける。機能が複雑になる
- **改善案**: 
  - **Primary Personaに集中**: 非エンジニアのマーケター/営業担当者
  - メッセージを「プログラミング不要」に統一
  - 高度な機能は「プロ向け」として後回し

**問題2: UI/UXとペルソナのミスマッチ**
- **現状**: 技術的な用語が多い（「APIキー」「認証情報」「実行履歴」）
- **問題**: Primary Personaには理解が難しい
- **改善案**:
  - 「APIキー」→「サービス連携キー」
  - 「認証情報」→「ログイン情報」
  - 「実行履歴」→「実行ログ」
  - ツールチップで用語を説明

## Phase 3: 競合リサーチ＆ポジショニング分析

### 3.1 競合プロダクトの特定

#### 直接競合（同じ課題を同じ方法で解決）

**1. Browser Use（直接競合）**
- **主要機能**: 自然言語でブラウザ自動化
- **価格体系**: オープンソース（無料）
- **ターゲットユーザー**: 開発者
- **強み**: 
  - オープンソースで無料
  - 技術的に優れている
- **弱み**: 
  - 開発者向けで、非エンジニアには難しい
  - UI/UXが未整備
  - スケジュール実行や履歴管理がない
- **UI/UXの特徴**: コマンドライン中心

**2. Playwright（間接競合）**
- **主要機能**: ブラウザ自動化フレームワーク
- **価格体系**: オープンソース（無料）
- **ターゲットユーザー**: 開発者、QAエンジニア
- **強み**: 
  - 強力な自動化機能
  - 大規模なコミュニティ
- **弱み**: 
  - プログラミング必須
  - 非エンジニアには使えない

#### 間接競合（同じ課題を別の方法で解決）

**3. Zapier**
- **主要機能**: サービス間の自動化（API連携中心）
- **価格体系**: 無料プランあり、有料は$20/月〜
- **ターゲットユーザー**: 非エンジニアのビジネスユーザー
- **強み**: 
  - 使いやすいUI
  - 豊富な連携（5000+サービス）
  - 明確な価値提案
- **弱み**: 
  - ブラウザ操作には対応していない
  - カスタムロジックが難しい
- **UI/UXの特徴**: ビジュアルワークフローエディタ、直感的

**4. Make (旧Integromat)**
- **主要機能**: サービス間の自動化（API連携中心）
- **価格体系**: 無料プランあり、有料は$9/月〜
- **ターゲットユーザー**: 非エンジニアのビジネスユーザー
- **強み**: 
  - 強力なワークフロー機能
  - ビジュアルエディタ
- **弱み**: 
  - ブラウザ操作には対応していない
  - 学習曲線がやや高い

**5. n8n**
- **主要機能**: ワークフロー自動化（API + ブラウザ操作）
- **価格体系**: オープンソース（無料）、クラウド版は$20/月〜
- **ターゲットユーザー**: 開発者、技術的なユーザー
- **強み**: 
  - オープンソース
  - ブラウザ操作に対応
  - 自己ホスト可能
- **弱み**: 
  - UI/UXが技術的
  - 非エンジニアには難しい
  - 自然言語対応なし

#### 代替手段

**6. Excel + VBA / Google Apps Script**
- **主要機能**: 手動操作の自動化
- **価格体系**: 無料〜$10/月
- **ターゲットユーザー**: エクセルユーザー
- **強み**: 
  - 既存ツールで追加コストなし
  - カスタマイズ可能
- **弱み**: 
  - プログラミング知識が必要
  - ブラウザ操作が難しい
  - メンテナンスが大変

### 3.2 ポジショニングマップ

```
                   高機能・高価格
                         ↑
                         │
                         │
                    [Make] [n8n]
                         │
                         │
    初心者向け ←────────┼────────→ プロ向け
                         │
                         │
                    [Zapier] [Browser Use]
                         │
                         │
                         ↓
                   低機能・低価格
```

**現在位置**: 
- X軸（初心者向け vs プロ向け）: やや初心者寄り（自然言語対応）
- Y軸（低機能・低価格 vs 高機能・高価格）: 中程度

**理想位置**: 
- X軸: より初心者寄り（UI/UXの改善）
- Y軸: 中程度を維持（ブラウザ操作という独自機能で差別化）

### 3.3 差別化ポイントの評価

#### 現状の独自性（Unique Value Proposition）

**強み**:
1. **自然言語でのブラウザ自動化**: Zapier/Makeにはない機能
2. **リアルタイムライブビュー**: 実行中の状態を可視化（Manus同等）
3. **動画から自動生成**: 革新的な機能
4. **API優先の自動化提案**: 効率的な方法を提案

**弱み**:
1. **ブランド認知度**: 競合に比べて知名度が低い
2. **連携サービスの数**: Zapier（5000+）に比べて少ない
3. **UI/UXの成熟度**: Zapier/Makeに比べて未成熟
4. **ドキュメント・サポート**: 競合に比べて不足

#### 競合に対する優位性/劣位性

| 機能 | Workflow Dashboard | Zapier | Make | n8n | Browser Use |
|------|-------------------|--------|------|-----|-------------|
| 自然言語対応 | ✅ | ❌ | ❌ | ❌ | ✅ |
| ブラウザ操作 | ✅ | ❌ | ❌ | ✅ | ✅ |
| ライブビュー | ✅ | ❌ | ❌ | ❌ | ❌ |
| 動画から生成 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 使いやすさ | ⚠️ | ✅ | ✅ | ⚠️ | ❌ |
| 連携サービス数 | ⚠️ | ✅ | ✅ | ⚠️ | ❌ |
| 価格 | ✅ | ⚠️ | ✅ | ✅ | ✅ |

#### 「これがあるからこのサービスを選ぶ」理由の明確化

**現状**: 明確ではない

**改善案**: 
- **メインメッセージ**: 「ブラウザ操作を自然言語で自動化。プログラミング不要、動画から自動生成、リアルタイム監視」
- **サブメッセージ**: 「Zapierではできないブラウザ操作を、コードを書かずに実現」
- **ターゲット**: 「毎日のデータ収集やレポート作成を自動化したいマーケター/営業担当者」

## Phase 4: グローバル展開チェック

### 4.1 ローカライゼーション

#### 現状の問題点

**問題1: 多言語対応が不十分**
- **影響度**: 高
- **現状**: UIが日本語と英語が混在（例：Sidebarは日本語、Tasksページは英語）
- **問題**: グローバル展開が不可能
- **改善案**:
  - **i18nフレームワークの導入**: `react-i18next`を使用
  - **優先順位**: 
    1. 英語（完全対応）
    2. 日本語（完全対応）
    3. 中国語（簡体字）
    4. スペイン語
    5. ドイツ語
  - **翻訳品質**: 専門の翻訳者を起用（機械翻訳はNG）
- **実装難易度**: 高
- **優先度**: P1
- **参考事例**: 
  - [Shopify](https://www.shopify.com/) - 20+言語対応
  - [Notion](https://www.notion.so/) - 多言語対応

**問題2: 文化的に不適切な表現**
- **影響度**: 中
- **現状**: 特に問題なし（確認が必要）
- **改善案**:
  - 各国の文化に配慮した表現を確認
  - 絵文字の使用を控える（文化的に受け入れられない場合がある）
  - 日付・時刻の表示形式を地域に合わせる

**問題3: 日付、通貨、単位の表示**
- **影響度**: 中
- **現状**: `History.jsx`で`toLocaleString('ja-JP')`がハードコードされている
- **問題**: 日本以外のユーザーには不適切
- **改善案**:
  - `Intl.DateTimeFormat`を使用してユーザーのロケールに合わせる
  - 通貨表示が必要な場合は`Intl.NumberFormat`を使用
  - 単位（ファイルサイズなど）もロケールに合わせる
- **実装難易度**: 低
- **優先度**: P2

### 4.2 アクセシビリティ

#### 現状の問題点

**問題1: WCAG 2.1基準への準拠度が不明**
- **影響度**: 高
- **現状**: アクセシビリティの確認が行われていない
- **問題**: 法的リスク、ユーザーベースの縮小
- **改善案**:
  - **色のコントラスト比**: 4.5:1以上を確保（AA基準）
  - **キーボードナビゲーション**: すべての機能にキーボードでアクセス可能
  - **フォーカスインジケーター**: 明確なフォーカス表示
  - **スクリーンリーダー対応**: `aria-label`、`aria-describedby`を適切に使用
  - **alt属性**: すべての画像に適切なalt属性
- **実装難易度**: 中
- **優先度**: P1
- **参考事例**: 
  - [WebAIM](https://webaim.org/) - アクセシビリティチェックリスト
  - [WAVE](https://wave.webaim.org/) - アクセシビリティ評価ツール

**問題2: スクリーンリーダー対応不足**
- **影響度**: 高
- **現状**: `Sidebar.jsx`で一部`aria-label`があるが、不十分
- **問題**: 視覚障がい者がサービスを利用できない
- **改善案**:
  - すべてのインタラクティブ要素に`aria-label`を追加
  - フォームに`aria-describedby`で説明を追加
  - エラーメッセージに`aria-live`を追加
  - ローディング状態に`aria-busy`を追加
- **実装難易度**: 中
- **優先度**: P1

**問題3: 色覚多様性への配慮不足**
- **影響度**: 中
- **現状**: 色だけで情報を伝えている箇所がある（例：ステータスバッジ）
- **問題**: 色覚障がい者が情報を理解できない
- **改善案**:
  - 色だけでなく、アイコンやテキストでも情報を伝える
  - コントラスト比を確保
  - 色覚シミュレーションツールで確認
- **実装難易度**: 低
- **優先度**: P2

### 4.3 パフォーマンス

#### 現状の問題点

**問題1: モバイル対応の品質が不明**
- **影響度**: 高
- **現状**: `Layout.jsx`でモバイルメニューはあるが、詳細な確認が必要
- **問題**: モバイルユーザーの体験が悪い可能性
- **改善案**:
  - **モバイルファーストデザイン**: モバイルから設計
  - **タッチターゲットサイズ**: 44x44px以上を確保
  - **レスポンシブ画像**: `srcset`を使用
  - **モバイルでのテスト**: 実際のデバイスでテスト
- **実装難易度**: 中
- **優先度**: P1
- **参考事例**: 
  - [Google Mobile-Friendly Test](https://search.google.com/test/mobile-friendly)

**問題2: 読み込み速度の最適化不足**
- **影響度**: 中
- **現状**: 画像最適化、コード分割の確認が必要
- **問題**: 各地域での読み込み速度が遅い可能性
- **改善案**:
  - **画像最適化**: WebP形式、遅延読み込み
  - **コード分割**: React.lazy、Suspenseを使用
  - **CDNの使用**: 静的アセットをCDNで配信
  - **パフォーマンス監視**: Lighthouse、WebPageTestで測定
- **実装難易度**: 中
- **優先度**: P2

### 4.4 信頼性・セキュリティ

#### 現状の問題点

**問題1: プライバシーポリシー、利用規約の明示不足**
- **影響度**: 高
- **現状**: フッターにリンクがない
- **問題**: GDPR/CCPA等の法的要件を満たせない
- **改善案**:
  - **プライバシーポリシー**: データの収集・使用・保存方法を明記
  - **利用規約**: サービスの利用条件を明記
  - **Cookieポリシー**: Cookieの使用を明記
  - **フッターにリンク**: すべてのページのフッターに配置
- **実装難易度**: 低
- **優先度**: P1
- **参考事例**: 
  - [Shopify](https://www.shopify.com/legal/privacy) - 明確なプライバシーポリシー

**問題2: GDPR/CCPA等への対応状況が不明**
- **影響度**: 高
- **現状**: 対応状況が不明
- **問題**: EU/カリフォルニア州での展開ができない
- **改善案**:
  - **データのエクスポート機能**: ユーザーが自分のデータを取得できる
  - **データの削除機能**: アカウント削除時にデータを完全削除
  - **同意の取得**: Cookie使用前に同意を取得
  - **透明性**: データの使用目的を明確に
- **実装難易度**: 高
- **優先度**: P1

## まとめ

Workflow Dashboardは、自然言語ブラウザ自動化という革新的なコンセプトと、ライブビュー機能という強力な差別化要素を持つ優れたプロダクトです。しかし、世界展開には以下の改善が不可欠です：

1. **初見ユーザー体験の根本的再設計**（ランディングページ、オンボーディング）
2. **多言語対応とローカライゼーション**（i18nフレームワークの導入）
3. **アクセシビリティとパフォーマンスの最適化**（WCAG 2.1 AA準拠、モバイル最適化）

これらの改善を段階的に実施することで、世界で愛されるプロダクトになる可能性を秘めています。
